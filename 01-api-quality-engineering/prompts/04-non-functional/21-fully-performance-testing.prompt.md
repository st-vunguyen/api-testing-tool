# Run Full Performance Testing

Use this prompt after `18-performance-scenarios.prompt.md` and `19-performance-collection.prompt.md` have already produced a workload model and performance artifact.

## Role

You are a **Performance Execution Engineer + API Reliability Analyst + Safe-Environment Gatekeeper**.

## Goal

Verify the generated performance assets, set up the required tooling, execute a safe and evidence-backed performance run, store raw outputs under the canonical raw-report path, and publish a curated report that clearly separates measured facts from assumptions and tooling limitations.

## Inputs

- `API_SPEC_PATH`
- `DOCS_PATHS`
- `OUTPUT_SLUG`
- `TARGET_STACK`: e.g. `postman-newman`, `k6`, `jmeter`, or another explicitly supported stack
- `TARGET_ENV`: `local | dev | staging | perf`
- `BASE_URL`: required for the actual run
- `AUTH_MODE`: `none | bearer | api-key` (descriptive only; no secrets)
- `EXECUTION_MODE`: `baseline | load-step | stress-lite` (default `baseline`)
- `SAFE_TO_RUN`: `yes | no`
- `MAX_DURATION_MINUTES`: default `15`
- `MAX_VUS_OR_ITERATIONS`: evidence-backed upper bound only

## Preconditions

Do not start execution until all of the following are true:

- `documents/api-quality-engineering/<OUTPUT_SLUG>/scenarios/performance-scenarios.md` exists
- `documents/api-quality-engineering/<OUTPUT_SLUG>/scenarios/performance-thresholds-and-assumptions.md` exists

- the target environment is explicitly safe for non-destructive performance execution

If any precondition is missing, refresh the missing asset first instead of inventing it.

## Guardrails

- This prompt is an execution template and must not store generated run output inside itself.
- Do not overwrite files under `.github/prompts/` or `.claude/prompts/`.
- Do not run destructive workloads against production.
- Do not commit real secrets, tokens, or current-value exported environments.
- If the selected stack is only partially suitable for performance testing, label the run as constrained and document the limitation explicitly.

## Anti-Hallucination Rules

- Do not invent throughput targets, latency SLAs, or error budgets.
- Do not claim load-test realism when the stack only supports lightweight iteration sampling.
- Do not claim bottlenecks without measured evidence from the run outputs.
- If system telemetry is unavailable, write `Needs infrastructure validation` instead of guessing server-side causes.

## Execution Method

### 1. Validate the generated performance plan

1. Read the outputs from prompts `18` and `19` in full.
2. Confirm the selected endpoints, workload model, thresholds, assumptions, and blockers are traceable.
3. If `TARGET_STACK=postman-newman`, classify the run as a constrained API workload check unless a dedicated performance runner artifact is also available.

### 2. Prepare the execution toolchain

1. Determine the actual runtime command for the chosen stack.
2. Create or refresh local runner scripts and environment examples.
3. Document prerequisites in `tools/api-quality-engineering/<OUTPUT_SLUG>/performance/README.md`.
4. Ensure raw outputs will land only under `tools/api-quality-engineering/<OUTPUT_SLUG>/reports/raw/performance/<run-slug>/`.

### 3. Enforce a safe run gate

Only continue if all are true:

- `SAFE_TO_RUN=yes`
- environment is isolated or explicitly approved for performance execution
- workload is non-destructive or cleanup is documented
- concurrency / iteration ceilings are documented and respected

If the gate fails, do not execute the run. Produce the report with `Blocked` status and explain why.

### 4. Execute in stages

Run in this order:

1. **Smoke calibration** — verify auth, variable binding, and endpoint reachability
2. **Baseline run** — collect measured latency, status distribution, and error rate
3. **Step-up load** — only if baseline is stable and the environment remains safe
4. **Stress-lite run** — optional and only when explicitly justified

After each stage:

- store raw outputs in the canonical raw-report folder
- update the curated report with facts, failures, and limitations
- stop escalation if error rate, safety concerns, or environmental instability exceeds the approved boundary

### 5. Publish evidence-backed findings

In the final report:

- separate measured results from assumptions
- separate performance findings from tooling limitations
- state whether the run is representative, constrained, or blocked
- list exact commands, environment shape, and workload parameters used
